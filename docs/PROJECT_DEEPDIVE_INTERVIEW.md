# OpenClaw 低频量化投研项目深度说明（面试版）

> 目标：你看完这份文档后，可以系统回答“项目架构、技术实现、指标设计、风控策略、工程落地、失败处理、可扩展性”相关问题。

---

## 1. 项目一句话概述

这是一个基于 OpenClaw 的 A 股低频量化投研 Agent：
通过定时抓取市场数据和财务数据，做多因子评分，生成带风险约束与变更说明的中文建议，并通过飞书通道推送日报/周报。

---

## 2. 为什么做这个项目（业务价值）

传统“看盘+拍脑袋”问题：
- 信息噪音大
- 同一结论重复提醒
- 无统一风控约束
- 结论不可追溯

本项目解决：
1. **结构化决策**：从数据到建议链路标准化。
2. **可解释性**：每条建议给出因子理由与置信度。
3. **风控优先**：组合约束优先于单票机会。
4. **降噪**：24h 去重，减少重复打扰。

---

## 3. 系统架构（四层）

### 3.1 数据层（MarketScanner）
输入：Tushare Token、股票池配置
输出：`market_snapshot.tushare.json`

核心字段：
- 行情：close、change_pct
- 因子输入：ret_1d/5d/20d、volatility_20d、volume_ratio_5d
- 财务输入：PE/PB、ROE、毛利率、资产负债率
- 质量：data_freshness_sec、source_health、snapshot_quality_score

### 3.2 分析层（SignalAnalyst）
输入：snapshot + signal_rules
输出：`signal_report.generated.json`

多因子打分：
- 趋势（trend）
- 动量（momentum）
- 估值（valuation）
- 波动（volatility）
- 回撤代理（drawdown）
- 财报质量（earnings_quality）

### 3.3 决策层（Advisor）
输入：signal_report + 历史建议
输出：`advice_actions.json`

能力：
- 24h 同结论同理由去重
- change_vs_last（与上次建议对比）
- 数据质量闸门降级（低质量不增持）

### 3.4 交付层（Notifier）
输入：日报/卡片 payload
输出：飞书消息（OpenClaw 通道）

能力：
- 卡片模板输出
- 推送状态回写 `push_job_state.json`
- 漏跑补偿（重启后检测）

---

## 4. 主流程时序（交易日 15:30）

1. 构建 market snapshot
2. 计算多因子信号
3. 计算组合风险约束
4. 生成建议并去重
5. 生成日报 markdown
6. 生成飞书卡片 payload
7. 更新推送状态
8. OpenClaw 通道推送

---

## 5. 关键算法细节（面试高频）

### 5.1 因子归一化
- 收益率因子：`normalize_ret(ret, lo=-8%, hi=8%)`
- 波动因子：`1 - volatility_20d / 0.06`（再 clamp 到 [0,1]）

### 5.2 估值打分
- PE 越低越好：`pe_s=(30-pe)/30`
- PB 越低越好：`pb_s=(5-pb)/5`
- 估值总分：`(pe_s + pb_s)/2`

### 5.3 财报质量打分
- ROE 高优：`roe/20`
- 毛利率高优：`gpm/50`
- 资产负债率低优：`(70-d2a)/70`
- 财报分：三项均值

### 5.4 总分模型
`score = Σ(weight_i * factor_i)`

默认权重：
- trend 0.25
- momentum 0.20
- valuation 0.15
- volatility 0.15
- drawdown 0.15
- earnings_quality 0.10

### 5.5 信号映射
- score >= 0.70 → increase
- 0.55~0.70 → hold
- 0.40~0.55 → observe
- <0.40 → reduce

## 6. 风险控制机制

### 6.1 数据质量闸门
当 `snapshot_quality_score < 60`：
- 强制降级，不输出增持
- 只给观察/风控建议

### 6.2 组合约束
- 单票 ≤ 20%
- 单行业 ≤ 35%
- 最大回撤容忍 -8%

### 6.3 去重降噪
- 同标的、同动作、同理由哈希、24h 内抑制推送

## 7. 工程实现亮点

1. **Schema 先行**：对关键 JSON 产物做结构约束。
2. **脚本解耦**：每步独立脚本，方便定位故障。
3. **状态可追踪**：push_job_state/advice_history 持久化。
4. **模板化输出**：日报、周报、卡片分离。
5. **版本治理**：配置文件与 README 同步更新并 push。

## 8. 你可以怎么回答“难问题”

### Q1：为什么不直接做端到端 LLM？
A：金融场景要可解释和可控，采用“规则+因子+模板”更适合生产稳定性和审计。

### Q2：数据质量差时如何避免误导？
A：有质量闸门，低质量自动降级为观察，不给激进建议。

### Q3：如何降低噪音？
A：24h 去重 + 无显著变化不重复提醒。

### Q4：如何证明模型在迭代？
A：每周产出 signal_eval_report，后续做置信度校准和参数更新。

### Q5：为什么选 OpenClaw？
A：OpenClaw 在“多工具编排 + 通道推送 + 任务定时 + 配置治理”上更适合做 Agent 工程化落地。

## 9. 当前局限与改进路线

局限：
- 部分 Tushare 高级接口权限可能不足
- 当前收益评估仍是轻量版（需接真实交易/回测链路）

路线：
1. 财务因子多源回退（Tushare→Akshare）
2. 周报自动推送闭环
3. 云端常驻部署
4. 置信度校准 + 回测归因
